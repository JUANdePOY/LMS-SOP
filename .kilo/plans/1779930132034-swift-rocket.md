# RBAC Implementation Plan: Sidebar Navigation & Feature Permissions

## Objective
Implement granular Role-Based Access Control (RBAC) for sidebar navigation and feature permissions based on user roles: `admin`, `admin_arsen`, `admin_group`, and `admin_squadron`.

## Current State Analysis
- All admin Airbase sub-items currently have `roles: ADMIN_ROLES` (all 4 admin variants)
- Backend has `getUserScopeFilter` but no role-specific menu filtering
- `ManageSquadrons.jsx` uses `requireAdmin` middleware (allows all admin roles)
- No restrictions on CRUD actions within pages

## Role Hierarchy Requirements
Based on role hierarchy:
- **admin**: Full access to all modules
- **admin_arsen**: Can manage ARCENs, Groups, Squadrons within their ARSEN; can promote reservists within their ARSEN
- **admin_group**: Can manage Squadrons within their Group; can promote reservists within their Group  
- **admin_squadron**: Can only view/manage their assigned Squadron; can promote reservists within their Squadron; **cannot** create/edit/delete squadrons

## Implementation

### 1. Update `client/src/config/menuItems.js`
Change `roles` property for Airbase children to specific role requirements:
- "Manage ARCENs": `roles: ['admin']` - Only super admin
- "Manage Groups": `roles: ['admin', 'admin_arsen']` - Super admin + ARCEN admin  
- "Manage Squadrons": `roles: ADMIN_ROLES` - All admin roles (will be view-only for admin_squadron)

The `filterMenuByRole` function already handles this correctly; it just needs the correct role arrays.

### 2. Update `client/src/App.jsx` (if needed)
Routes are already protected; no changes needed if menu filtering works.

### 3. Update `ManageSquadrons.jsx`
- Get user role from `useAuth()` 
- Hide "Add Squadron" button when `role === 'admin_squadron'`
- Disable Edit/Delete buttons in detail modal when `role === 'admin_squadron'`
- UI uses `getSquadrons()` API which is already scoped via `getUserScopeFilter` middleware

### 4. Update `ManageGroups.jsx`
- Hide "Add Group" button for all roles (backend `requireAdmin` allows all admin roles but mutations are scoped)
- Actually: backend uses `requireAdmin` which allows all admin roles, but mutations need scope validation for `admin_arsen`

### 5. Update Backend `server/routes/squadron.js`
Current issue: `requireAdmin` allows all admin roles including `admin_squadron` to create/edit/delete, but `admin_squadron` should only have read access.

Options:
- A) Create `requireAdminArsenGroup` middleware to exclude `admin_squadron` from mutations
- B) Add scope validation in handlers to reject operations outside user's scope

Recommendation: Use Option A - create scoped middleware for mutations.

### 6. Fix Backend `server/routes/groups.js`  
Current: `requireAdmin` allows all admin roles to create/edit/delete groups
Issue: `admin_squadron` can see this route but shouldn't be able to modify
Fix: `admin_arsen` should be able to create groups within their ARSEN

### 7. Backend `server/routes/arsents.js`
Already correct - `requireAdmin` restricts mutations to super admin only.
