# Scoped User Management Module Plan

## Objective
Implement scoped user management for `admin_arsen` and `admin_group` roles, allowing them to create and assign users within their defined scope while enforcing the constraints:
- `Admin_Arcen` can manage only their assigned Arcen, Groups, and Squadrons (and users within that scope).
- `Admin_Group` can manage only their assigned Group (and users within that scope).
Both roles are strictly prohibited from creating or managing additional Arcen/Group entities beyond their scope.

## Current System Analysis
- **Database Schema**: The `users` table includes `role`, `scope_arsen_id`, `scope_group_id`, `scope_squadron_id`.
- **Roles**: `admin` (super admin), `admin_arsen`, `admin_group`, `admin_squadron`, `reservist`.
- **Scoping**: Defined in RBAC matrix; unit admins see only records matching their scope.
- **Existing Endpoints**: 
  - `server/routes/settings.js` is restricted to `requireSuperAdmin` (only `admin` can manage users/roles/scopes).
  - Detail endpoints for arsents/groups/squadrons lack scope enforcement (security gaps).
  - List endpoints have partial scope enforcement but may not override user-provided filters correctly.

## Design
### 1. Middleware: Scoped User Management
Create `server/middleware/scopedUserManagement.js` with:
- `canManageUser(requestingUser, targetUser)`: Returns boolean if requesting user can manage target user based on:
  - **Super Admin (`admin`)**: Can manage any user.
  - **Admin_Arcen (`admin_arsen`)**:
    - Can manage users whose scope is within their Arcen:
      - `targetUser.scope_arsen_id === requestingUser.scope_arsen_id`
      - OR `targetUser` is a reservist assigned to a group/squadron under the Arcen.
      - OR `targetUser.scope_group_id` belongs to a group under the Arcen.
      - OR `targetUser.scope_squadron_id` belongs to a squadron under the Arcen.
    - Cannot manage users with `role = 'admin'` (super admin) or `admin_arsen` with a different `scope_arsen_id`.
    - Can assign roles: `admin_group`, `admin_squadron`, `reservist` (and optionally `admin_arsen` for same Arcen?).
    - Can assign scopes only within their Arcen:
      - `scope_arsen_id`: only their own Arcen ID.
      - `scope_group_id`: only groups under their Arcen.
      - `scope_squadron_id`: only squadrons under their Arcen.
  - **Admin_Group (`admin_group`)**:
    - Similar logic restricted to their Group:
      - Can manage users whose scope is within their Group:
        - `targetUser.scope_group_id === requestingUser.scope_group_id`
        - OR `targetUser` is a reservist assigned to a squadron under the Group.
        - OR `targetUser.scope_squadron_id` belongs to a squadron under the Group.
      - Cannot manage users with `role = 'admin'` or `admin_arsen` (different scope) or `admin_group` with different `scope_group_id`.
      - Can assign roles: `admin_squadron`, `reservist` (and optionally `admin_group` for same Group?).
      - Can assign scopes only within their Group:
        - `scope_group_id`: only their own Group ID.
        - `scope_squadron_id`: only squadrons under their Group.
  - **Admin_Squadron (`admin_squadron`)**:
    - Can manage users within their Squadron (similar logic).
    - Can assign roles: `reservist` only.
    - Can assign scopes only to their own Squadron ID.
  - **Reservist**: Cannot manage any users.

### 2. API Endpoints (Modify `server/routes/settings.js`)
- Change middleware from `requireSuperAdmin` to `requireScopedUserManagement` for user management routes.
- **GET `/api/settings/users`**:
  - Filter users by requesting user's scope using a helper function `getUserScopeFilterForUsers(requestingUser)`.
  - Return only users that the requesting user can manage.
- **PUT `/api/settings/users/:id/role`**:
  - Validate that `req.params.id` (target user) is manageable by requesting user.
  - Validate that the new `role` is assignable by requesting user.
  - Validate that new scope fields (`scope_arsen_id`, `scope_group_id`, `scope_squadron_id`) are within requesting user's scope.
  - If role is `reservist`, clear all scope fields (as reservists have no scope).
  - Update `users` and `user_role_history` tables.
- **Additional Endpoints** (if needed):
  - `POST /api/settings/users`: Create a new user (with scoped validation).
  - `DELETE /api/settings/users/:id`: Soft delete/deactivate a user (if in scope).

### 3. Security Fixes for Existing Endpoints
While out of scope for this module, note that the following should be addressed in a separate task:
- Detail endpoints (`/api/arsens/:id`, `/api/groups/:id`, `/api/squadron/:id`) must enforce scope.
- List endpoints must ensure scope overrides user-provided filters correctly (already partially implemented).

## Implementation Steps
1. **Create Middleware**:
   - Write `server/middleware/scopedUserManagement.js` with `canManageUser` logic and `requireScopedUserManagement` middleware.
2. **Update Settings Route**:
   - Modify `server/routes/settings.js`:
     - Import new middleware.
     - Replace `requireSuperAdmin` with `requireScopedUserManagement` for user management routes.
     - Implement scoping in GET `/users`.
     - Implement validation and scoping in PUT `/users/:id/role`.
     - Add POST/DELETE endpoints if required.
3. **Database**: No schema changes needed; use existing tables.
4. **Frontend** (out of scope for this plan but note):
   - Update React Settings page to show user management UI to scoped admins.
   - Restrict available roles/scopes in forms based on current user's scope.
5. **Testing**:
   - Manual test with super admin, admin_arsen, admin_group, admin_squadron, reservist users.
   - Verify scoping constraints and prohibition of creating/managing additional Arcens/Groups.

## Verification Criteria
- [ ] Super admin retains full user management capabilities.
- [ ] Admin_arsen can only manage users within their Arcen (cannot manage users outside, cannot create users with role `admin` or `admin_arsen` for different Arcen).
- [ ] Admin_arsen can assign roles `admin_group`, `admin_squadron`, `reservist` within their Arcen.
- [ ] Admin_arsen cannot assign scopes outside their Arcen.
- [ ] Admin_group can only manage users within their Group (cannot manage users outside, cannot create users with role `admin`, `admin_arsen`, or `admin_group` for different Group).
- [ ] Admin_group can assign roles `admin_squadron`, `reservist` within their Group.
- [ ] Admin_group cannot assign scopes outside their Group.
- [ ] Admin_squadron can only manage users within their Squadron (limited to reservists).
- [ ] Reservist cannot manage any users.
- [ ] All role changes are logged in `user_role_history` and audit logs.

## Risks and Mitigations
- **Risk**: Incorrect scoping logic leading to privilege escalation.
  - **Mitigation**: Implement comprehensive unit tests for `canManageUser` and manual testing with various role/scope combinations.
- **Risk**: Breaking existing super admin functionality.
  - **Mitigation**: Ensure super admin bypasses all scoping checks.
- **Risk**: Performance impact from complex scoping queries.
  - **Mitigation**: Optimize queries; use joins where possible.

## Frontend Considerations (Separate Task)
- Update `client/src/pages/Settings.jsx`:
  - Show user management tabs only if user can manage users (per `canManageUser` logic).
  - Populate role/scope dropdowns based on current user's scope.
  - Prevent assignment of invalid roles/scopes.
- Update `client/src/config/menuItems.js` to show Settings link to scoped admins if they can manage users.
