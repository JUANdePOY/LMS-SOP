# Technical Specification: User Profile Management Feature

## Overview

Enable authenticated users (reservists) to update their own profile information while ensuring the `trainings` field remains strictly read-only for non-admin users. Administrators retain full CRUD access to training data.

---

## 1. Database Schema Assessment

**Result:** No schema changes required. Existing tables already support this feature:

| Table | Relevant Columns |
|-------|-----------------|
| `users` | `id`, `email`, `password_hash`, `role` |
| `reservists` | `user_id`, `first_name`, `last_name`, `rank`, `phone_number`, `address`, `emergency_contact_*`, `blood_type`, `address`, `is_active` |
| `trainings` | `id`, `title`, `start_datetime`, `end_datetime`, `status` (managed via `internal_training_participants`) |
| `internal_training_participants` | `training_id`, `reservist_id`, `squadron_id` (link table, admin-writable only) |

The `trainings` concept is a **computed relationship** via `internal_training_participants`, not a column on the `reservists` table. This design already enforces the separation between profile data and training assignments.

---

## 2. API Endpoint Structure

### New Endpoint — Self-Service Profile Update

```
PUT /api/reservists/my/profile
Authorization: Bearer <JWT>
```

**Request body fields (all optional, whitelisted):**

| Field | Type | Description |
|-------|------|-------------|
| `first_name` | string | 2-100 chars |
| `last_name` | string | 2-100 chars |
| `rank` | string | Military rank |
| `phone_number` | string | Contact number |
| `address` | text | Home address |
| `date_of_birth` | date | ISO 8601 |
| `sex` | enum | Male / Female / Other |
| `blood_type` | enum | A+ / A- / B+ / AB+ / AB- / O+ / O- / Unknown |
| `emergency_contact_name` | string | Emergency contact |
| `emergency_contact_phone` | string | Emergency contact number |
| `emergency_contact_address` | text | Emergency contact address |
| `civil_status` | enum | Single / Married / Widowed / Separated / Divorced |
| `citizenship` | string | Default: Filipino |

**Strictly excluded (must never be accepted from non-admin users):**
- `trainings` / `training_id` / `internal_training_participants`
- `is_active`
- `service_number`
- `user_id`
- `reserve_status` (admin-only field)
- `qr_code`

**Success response (200):**
```json
{
  "status": "success",
  "message": "Profile updated successfully",
  "data": { /* updated reservist row */ }
}
```

**Error responses:**
- `401` — No valid token
- `403` — User is not a reservist (admin must use `PUT /api/reservists/:id`)
- `404` — No reservist profile found for user
- `409` — Training fields attempted in request body (with `code: "FORBIDDEN_TRAINING_UPDATE"`)

**Protection against training field mutation:**
If the request body contains any of the training-related keys (e.g., `training_id`, `trainings`, `internal_training_participants`), reject with `409 Conflict` before processing. Do **not** silently ignore — explicit denial is an audit signal.

### Existing Endpoints (No Change Needed)

| Method | Path | Role Required | Purpose |
|--------|------|--------------|---------|
| `GET` | `/api/reservists/my/profile` | Authenticated | Read own profile |
| `GET` | `/api/reservists/my/trainings` | Authenticated | Read own training history |
| `GET` | `/api/reservists/my/attendance` | Authenticated | Read own attendance |
| `GET` | `/api/reservists/my/readiness` | Authenticated | Read own readiness |
| `GET` | `/api/auth/profile` | Authenticated | Read users.email/role |
| `PUT` | `/api/auth/profile` | Authenticated | Update **users** email/password only |
| `PUT` | `/api/reservists/:id` | `requireAdmin` | Admin updates any reservist record |
| `POST` | `/api/internal` | `requireAdmin` | Admin creates training |
| `PATCH` | `/api/internal/:id` | `requireAdmin` | Admin updates training |

## 3. Implementation File Changes

### 3.1 `server/routes/reservists.js`

Add new route after existing self-service routes (~line 2070):

```javascript
const TRAINING_FORBIDDEN_FIELDS = [
  'training_id', 'trainings', 'internal_training_participants',
  'external_training_attachments', 'training_registrations'
];

router.put(
  '/my/profile',
  authenticateToken,
  [
    body('first_name').optional().trim().isLength({ min: 2, max: 100 }),
    body('last_name').optional().trim().isLength({ min: 2, max: 100 }),
    body('rank').optional().trim(),
    body('phone_number').optional().trim(),
    body('address').optional(),
    body('date_of_birth').optional().isISO8601(),
    body('sex').optional().isIn(['Male', 'Female', 'Other']),
    body('blood_type').optional().isIn(['A+','A-','B+','B-','AB+','AB-','O+','O-','Unknown']),
    body('emergency_contact_name').optional().trim(),
    body('emergency_contact_phone').optional().trim(),
    body('emergency_contact_address').optional(),
    body('civil_status').optional().isIn(['Single','Married','Widowed','Separated','Divorced']),
    body('citizenship').optional().trim(),
  ],
  async (req, res) => {
    try {
      const errors = validationResult(req);
      if (!errors.isEmpty()) {
        return res.status(400).json({ status: 'error', errors: errors.array() });
      }

      // Reject training-related fields explicitly
      const bodyKeys = Object.keys(req.body);
      const forbidden = bodyKeys.filter(k => TRAINING_FORBIDDEN_FIELDS.includes(k));
      if (forbidden.length > 0) {
        logAudit({
          user_id: req.user.id,
          action: 'profile_update.attempted_training_field',
          entity_type: 'reservist',
          entity_id: req.user.id,
          new_values: { forbidden_fields: forbidden },
          ip_address: getClientIp(req),
          user_agent: req.headers['user-agent']
        });
        return res.status(409).json({
          status: 'error',
          message: 'Training data cannot be modified through profile endpoint',
          code: 'FORBIDDEN_TRAINING_UPDATE'
        });
      }

      const [reservist] = await db.query(
        'SELECT id, user_id FROM reservists WHERE user_id = ?',
        [req.user.id]
      );

      if (reservist.length === 0) {
        return res.status(404).json({
          status: 'error',
          message: 'Reservist profile not found',
          code: 'NOT_FOUND'
        });
      }

      const reservistId = reservist[0].id;
      const allowedFields = [
        'first_name', 'last_name', 'rank', 'phone_number', 'address',
        'date_of_birth', 'sex', 'blood_type',
        'emergency_contact_name', 'emergency_contact_phone', 'emergency_contact_address',
        'civil_status', 'citizenship'
      ];

      const updateFields = {};
      allowedFields.forEach(field => {
        if (req.body[field] !== undefined) {
          updateFields[field] = req.body[field];
        }
      });

      if (Object.keys(updateFields).length === 0) {
        return res.status(400).json({
          status: 'error',
          message: 'No valid fields to update',
          code: 'NO_UPDATES'
        });
      }

      const updateColumns = Object.keys(updateFields).map(col => `\`${col}\` = ?`).join(',');
      const values = [...Object.values(updateFields), reservistId];

      await db.query(`UPDATE reservists SET ${updateColumns} WHERE id = ?`, values);

      logAudit({
        user_id: req.user.id,
        action: 'reservist.profile_updated',
        entity_type: 'reservist',
        entity_id: reservistId,
        old_values: null,
        new_values: updateFields,
        ip_address: getClientIp(req),
        user_agent: req.headers['user-agent']
      });

      const [updated] = await db.query(
        'SELECT * FROM reservists WHERE id = ?',
        [reservistId]
      );

      res.json({
        status: 'success',
        message: 'Profile updated successfully',
        data: updated[0]
      });
    } catch (error) {
      console.error('Error updating own profile:', error);
      res.status(500).json({
        status: 'error',
        message: 'Failed to update profile',
        code: 'UPDATE_ERROR'
      });
    }
  }
);
```

## 4. Security Measures

### 4.1 Authentication & Authorization
- All endpoints use `authenticateToken` — verifies JWT, confirms user exists and `is_active = TRUE`
- The self-service update derives ownership from `req.user.id → reservists.user_id`, preventing IDOR
- `requireAdmin` still guards all training mutation routes (`POST/PATCH/DELETE /api/internal/*`)

### 4.2 Input Validation
- Whitelist-based field selection — only allowed fields are extracted from `req.body`
- Training-related keys are explicitly rejected with `409 Conflict`
- All string/date fields use express-validator rules
- Password changes remain on `PUT /api/auth/profile` (bcrypt with 12 salt rounds — current `app/auth.js:SALT_ROUNDS`)

### 4.3 Audit Logging
Log the following events to `audit_logs`:
1. **Every successful profile update** — `action: 'reservist.profile_updated'`, include `old_values` and `new_values`
2. **Attempted training field update** — `action: 'profile_update.attempted_training_field'`, log the forbidden keys and IP
3. **No changes detected** — reject early with `400`, no DB hit, no audit noise

### 4.4 Rate Limiting (Recommended)
Add lightweight rate limiting to the `/my/profile` endpoint (e.g., `express-rate-limit`):
- Max 10 profile updates per 15 minutes per user
- Prevents automated PII scraping or bulk data exfiltration attempts

### 4.5 Defense Against Mass Assignment
The allowed-fields whitelist pattern already prevents mass assignment attacks. Never use `...req.body` or spread all body fields into the UPDATE query.

### 4.6 Sensitive Data Exposure
- Do NOT return `password_hash` in any profile GET response
- The existing `GET /api/reservists/my/profile` already omits `password_hash` (it's on the `users` table, joined via `u.password_hash` not selected in current query — verify and add explicit exclusion if needed)
- `qr_code` is returned for scannning purposes — this is intentional for attendance

## 8. Bug Fix: Password Change Returns 404 "User not found"

**Root Cause:** Property name mismatch between `authMiddleware` and `auth.js` routes.

| Location | Property Used |
|---|---|
| `server/middleware/auth.js` | `req.user.id` (from `users.id` DB column) |
| `server/routes/auth.js` | `req.user.userId` (from JWT payload) |
| All other routes (reservists, supplies, issuances, assignments) | `req.user.id` ✓ |

The JWT was generated with `userId: user.id`, but the middleware renames it to `id`. Only `auth.js` uses the stale `req.user.userId` name, causing `undefined` to be passed to `WHERE id = ?` → no rows → 404.

### Fix

In `server/routes/auth.js`, replace all 9 occurrences of `req.user.userId` with `req.user.id`:
- Line 142: `logAudit` user_id
- Line 145: `logAudit` entity_id
- Line 227: register endpoint user_id
- Line 272: GET /profile query param
- Line 333: PUT /profile SELECT query
- Line 392: PUT /profile UPDATE params push
- Line 396: PUT /profile logAudit user_id
- Line 399: PUT /profile logAudit entity_id
